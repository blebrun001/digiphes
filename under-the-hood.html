IPHES-CERCA Collections Digitization

Under the hood: understand the architecture in 5 minutes

The project follows a "100% graph" approach: data are organized in RDF, structured with CIDOC-CRM and its extensions, then explored through a web interface connected to a SPARQL endpoint.

This page follows a pedagogical path: from concept to implementation, ending with practical constraints observed in the MWE.

1) Why use a graph database (RDF)?

RDF stores information as subject-predicate-object triples. This format aligns naturally with CIDOC-CRM, where classes become nodes and properties become explicit relations.

Main benefits: fine-grained semantic modeling, interoperability (ARIADNE, Europeana, EOSC), and possible automatic inference in some cases.

Main tradeoff: higher technical complexity for day-to-day laboratory operations.

2) How CIDOC-CRM becomes machine-actionable

In RDF implementation, CIDOC-CRM acts as a formal vocabulary. Objects, events, and places are RDF classes; CRM links become RDF relations between these classes.

Simplified example: a lithic tool, its stratigraphic unit, excavation, traceological observation, and digitization activity become nodes. "Found by", "consists of", "observes", and "documents" become explicit links.

Result: RDF does not introduce a competing model; it materializes CRM reasoning as a machine-processable graph.

3) Chain of evidence (from field to dissemination)

The chain of evidence describes the progressive construction of scientific information, from observable material fact to interpretation and published data.

Each intermediate step remains traceable: linked, dated, attributed, and revisable assertions. It is not a single definitive proof, but a documented sequence.

4) The MWE: concrete technical architecture

The Minimum Working Example validates this end-to-end approach with Apache Jena Fuseki (SPARQL server) and TDB2 (persistent RDF storage).

Data are modeled in Turtle and queried with SPARQL. Infrastructure is containerized with Docker Compose for reproducible deployment of both triplestore and application layer.

Important: MWE data are placeholders and do not represent validated production data.

5) Semantic layer: ontologies, vocabularies, and graph separation

The ontological core is based on CIDOC-CRM v7.1.3, CRMarchaeo, and CRMsci, loaded into the triplestore in RDF/OWL.

Reference resources (UBERON, Darwin Core, Getty AAT, PeriodO, Neotoma, OME) are integrated as separate named graphs to preserve provenance and versioning.

The architecture distinguishes TBox (ontologies), SKOS (controlled vocabularies), and ABox (instance data). This separation supports reasoning, SHACL validation, and model evolution without app code changes.

6) Interface and validation

The Node.js / Express interface acts as a proxy to SPARQL and provides entity exploration, predefined queries, and graph visualizations.

SHACL is used to validate structural and semantic constraints in RDF graphs.

7) Key takeaways from the MWE

1. Implementation is demanding: it requires specialized skills and multiple technical components.

2. Conceptual framing must be rigorous: reference model, ontologies, and vocabularies must be precisely defined.

3. UX is harder to design, because a holistic graph approach increases information types, options, and interaction patterns.
